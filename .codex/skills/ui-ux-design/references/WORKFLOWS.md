## ワークフロー集

最終レポートは templates/UIUX_AUDIT_REPORT.md に沿う。
ここでは、現場で迷いがちな「どの順で見るか」を固定化する。

## 1. Web UI監査（URLがある）

入力
- URL
- 主要フロー（例: 検索→詳細→購入）
- 対象（デスクトップ中心、モバイル中心）

手順
- Chrome DevTools MCPで対象を開く
- スナップショットで構造と主要導線（CTA、ナビ、フォーム）を特定する
- 複数ビューポートで崩れと到達性を確認する
- 重要箇所のみ証跡としてスクリーンショットを取る
- ConsoleとNetworkを確認し、UI不具合や通信失敗を裏取りする
- 体感速度が課題ならパフォーマンストレースを取り、原因を推定する

出力
- 重大課題（P0/P1）を先にまとめる
- 画面間のズレを「統一ルール」として提案する
- 必要ならトークンとコンポーネント規格へ落とす

## 2. Web フォーム監査（登録、決済、申請）

目的
- 入力負担と離脱要因を減らす

手順
- スナップショットで入力項目、必須、説明文、エラー領域を特定する
- テスト環境やダミー入力が可能なら、入力して挙動を観察する
- 観察ポイント
  - エラーの出るタイミング
  - 何を直せば通るかが分かるか
  - 入力保持がされるか
  - 送信中の状態表示と二重送信防止
- 代替案を2つ用意する
  - 低コスト案（文言とレイアウト中心）
  - 中コスト案（入力の削減や構造変更）

出力
- 減らせる入力、自動化できる入力、後回しにできる入力を分類して提案する
- エラー文は「行動が分かる文」へ書き換える

## 3. Web 一貫性監査（複数画面）

目的
- UIのズレを減らし、学習コストと認知負荷を下げる

手順
- 代表画面を3〜7枚選び、それぞれスナップショットを取る
- 差分を抽出する
  - 見出し階層
  - 余白スケール
  - ボタンの種類と命名
  - フォームのラベル、ヘルプ、エラーの配置
  - 状態（loading/empty/error）の有無
- 差分を「ルール」へ落とす
  - トークン案
  - コンポーネント仕様案
  - 適用優先順（影響と頻度で決める）

出力
- 統一ルール一覧
- 置換戦略（どの部品から直すか）
- 受け入れ条件（統一が達成された判定）

## 4. Mobile UI監査（既存アプリ）

入力
- 対象アプリ
- 対象端末（画面サイズ）
- 主要フロー

手順
- Mobile MCPで端末とアプリを特定し起動する
- 各ステップでスクリーンショットと画面要素リストを取得する
- 観察ポイント
  - どこを押せばいいかが分かるか
  - 押した後のフィードバックがあるか
  - 戻るが一貫しているか
  - 入力支援（キーボード種別、補完、フォーマット）
  - Safe Areaと固定要素の干渉
- 縦横回転が可能なら崩れも確認する

出力
- 重大課題（迷い、行き止まり、誤操作）を優先度順に提示する
- iOSとAndroidで代替案が変わる場合は両案を併記する

## 5. 回帰確認（修正後）

目的
- 直したことで別の崩れが出ていないかを短時間で確認する

手順
- 代表画面を中心に、共通部品（ヘッダ、ナビ、ボタン、フォーム）を確認する
- Consoleやエラーの増加がないかを見る（Web）
- 重点確認
  - フォーカス
  - エラー表示
  - 主要CTAの到達性
  - レイアウト崩れ
