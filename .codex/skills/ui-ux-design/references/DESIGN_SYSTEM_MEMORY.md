## デザインシステムメモリ運用

目的
- 監査や改善提案で決めた「統一ルール」を、次の作業でも再利用できる形に固定する
- 画面単位の指摘を、部品単位の仕様へ落とし、ブレを減らす

## 推奨ファイル

- プロジェクト直下に design-system-memory.md を作る
- または design/ フォルダ配下に置く

## 何を保存するか

- トークン
  - 色（意味トークン）
  - タイポ（本文、見出し、注釈）
  - 余白（スケールと密度）
  - 角丸、影、モーション
- コンポーネント仕様
  - Button, TextField, Select, Dialog, Toast, Table など
  - バリアントと状態
  - 禁則（やってはいけない使い方）
- パターン
  - フォーム
  - 空状態
  - エラー
  - 一覧→詳細
- 文言方針
  - ラベルの動詞
  - エラー文の型
  - トーンの統一

## どう更新するか

- 新しい画面を作ったら、使ったコンポーネントが既存仕様に沿っているかチェックする
- 仕様から外れた場合は、必ず理由を書く
  - 例: 新しい要件で既存バリアントが不足した
- 例外は増やしすぎない
  - 例外が増えると、結局統一できなくなる

## 監査での使い方

- 画面を見て「ズレ」を見つけたら、まずメモリのルールと照合する
- メモリが不足している場合は、ルール提案を先に行う
- レポートには以下を残す
  - 既存ルールの逸脱
  - ルールの追加が必要か
  - 例外として許すか
